# 企业级资产管理系统（EAM / ITAM）领域知识库
> 版本：v1.0  
> 适用范围：企业 IT 设备资产（硬件/软件/云订阅/耗材）与通用企业资产（设施/工装/仪器等）  
> 目标：作为规范驱动开发流程中的“领域知识输入”，用于提升 Specify / Plan / Blueprint / Tasks / Implement 的生成质量与可追溯性。

---

## 1. 背景与目标（Why）

企业资产管理系统的核心目标，是在**资产全生命周期**内做到：
- **可见性**：资产在哪里、谁在用、什么状态、价值多少
- **可控性**：分配/借用/维修/报废过程可控，有审批、有规则、有审计
- **可追溯性**：资产历史操作链路不可篡改、可审计、可复盘
- **成本优化**：减少重复采购、降低丢失率、提升利用率、控制维修成本
- **风险降低**：降低安全风险（数据泄露、违规使用）、合规风险（审计缺口）

在 IT 资产管理（ITAM）实践中，一般强调端到端生命周期自动化管理硬件、软件、云资产，从而“优化成本、降低风险”。:contentReference[oaicite:0]{index=0}  
在资产管理标准（ISO 55000）视角下，资产管理强调跨生命周期实现价值，并综合考虑成本与风险。:contentReference[oaicite:1]{index=1}

## 2. 资产范围与分类（What）

企业级资产管理通常覆盖以下资产类型（可按组织需要启用/禁用）：

### 2.1 硬件资产（Hardware）
- 笔记本、台式机、显示器、手机/平板
- 服务器、存储、网络设备（交换机/路由器/防火墙）
- 外设与配件（扩展坞、键鼠、摄像头）
- 消耗/耗材（网线、适配器、工位配件等）

硬件资产管理通常需要“跟踪与管理所有硬件资产的端到端生命周期，并支持自动化与日常任务流”。:contentReference[oaicite:2]{index=2}

### 2.2 软件资产（Software / License）
- 商业软件许可（Office、IDE、设计软件、数据库等）
- 许可证类型：按设备 / 按用户 / 并发 / 订阅制
- 软件安装合规性、超量使用风险、续费提醒

### 2.3 云资产与订阅（Cloud / SaaS）
- 云资源（云主机、对象存储、数据库实例）
- SaaS 订阅（协作、工单、会议、知识库）
- 账号与权限关联、成本中心分摊、闲置识别

### 2.4 通用企业资产（EAM 视角）
- 设施设备（空调、电梯、发电机）
- 生产工装、测量仪器、实验设备
- 车辆、办公家具等

EAM 通常更强调维护管理、资产绩效与成本分析，通过维护计划延长资产寿命、降低总成本。:contentReference[oaicite:3]{index=3}

## 3. 企业级能力地图（Capabilities）

你可以把“企业级资产管理系统”能力拆成 9 个模块群：

1) **资产台账与资产模型（Asset Registry & Model）**  
2) **采购与合同（Procurement & Contract）**  
3) **入库与库存（Receiving & Inventory）**  
4) **领用/借用/归还（Assignment/Borrow/Return）**  
5) **维修与保养（Maintenance / Work Order / SLA）**  
6) **盘点与审计（Audit / Inventory Check）**  
7) **报废与处置（Retire & Disposal）**  
8) **财务与折旧（Finance / Depreciation / Chargeback）**  
9) **集成与自动化（Integration / Automation / Security）**

资产生命周期常见表达是：从**请求/采购 → 到货/入库 → 部署/使用 → 维护 → 退役/处置**（也可能进一步细化）。:contentReference[oaicite:4]{index=4}

## 4. 核心概念与数据对象（Domain Model）

> 下面是企业级资产管理最常见、最稳定的一组核心实体（建议作为你 Spec 的“领域词典”输入）

### 4.1 Asset（资产实例）
代表一台具体的设备/实例，是资产管理的核心对象。

**关键字段（建议至少具备）：**
- `assetId`：系统内唯一 ID（UUID）
- `assetTag`：内部资产编号（贴纸/二维码）
- `serialNumber`：序列号（硬件唯一标识）
- `assetType`：资产类型（Laptop / Monitor / Server / Network / Software / Cloud）
- `modelId`：关联资产型号（AssetModel）
- `ownerOrg`：资产归属组织（集团/子公司/法人）
- `costCenter`：成本中心（部门/项目）
- `purchaseInfo`：采购信息（供应商、采购价、币种、日期）
- `warrantyInfo`：保修信息（开始/结束、保修条款）
- `status`：当前状态（见第 5 节）
- `location`：位置（仓库/机房/城市/楼层/工位）
- `custodian`：当前保管人（员工/部门）
- `securityLevel`：安全等级（是否涉密/是否需加密盘/是否必须 MDM）
- `lastSeenAt`：最近一次“确认存在”的时间（盘点/扫描/发现系统）
- `retireReason`：退役原因（如报废/丢失/被盗/转卖）

### 4.2 AssetModel（资产型号/规格）
描述“这一类设备”的标准配置，用于采购与分配策略。
- 品牌、型号、配置（CPU/RAM/SSD）
- 标准价格区间、推荐适用岗位（研发/销售/设计）
- 标准保修策略、可选配件列表

### 4.3 InventoryItem / Stock（库存项）
资产在仓库/网点中的库存状态（特别适用于耗材与可替换配件）。
- `warehouseId`、`binLocation`
- `qtyOnHand` / `qtyReserved`
- 库存预警阈值（min/max）

### 4.4 Assignment（领用/分配记录）
记录“谁在何时持有此资产”，是审计最重要的链路之一。
- `assignedTo`（员工）
- `assignedAt`、`returnedAt`
- `assignmentType`（长期领用/临时借用）
- `expectedReturnAt`（借用才有）
- `approvalId`（若需要审批）
- `handoverChecklist`（交付检查项：外观、配件、擦除确认）

### 4.5 BorrowRequest（借用/领用申请）
用于审批流的起点（尤其是企业级系统常见强审批）。
- 借用原因、期限、目标资产类型
- 期望配置（或指定资产）
- 归还时间与超期规则

### 4.6 WorkOrder / MaintenanceTicket（维修工单）
用于资产维修、保养、报修处理。
- 故障描述、优先级、SLA
- 维修动作、备件更换、成本记录
- 结果：修复 / 无法修复 / 建议报废

### 4.7 Contract / Warranty（合同/保修）
企业级系统很依赖合同维度的统一管理：
- 框架协议、供应商合同
- 延保/维保服务包
- 采购订单与到货批次关联

### 4.8 AuditLog（审计日志）
所有关键动作必须不可篡改记录：
- 谁（操作者）在何时对哪个对象做了什么变更
- 变更前/后字段差异
- 触发来源（UI/API/批量导入/自动化规则）

### 4.9 FinanceRecord（财务记录）
折旧、摊销、资本化、费用化、资产净值等。
- 折旧方法：直线法/加速折旧
- 残值率、折旧年限、当前净值
- 成本中心分摊、项目归集

## 5. 生命周期状态机（State Machine）

企业级系统建议把资产状态拆成更细，确保流程可控且可审核。

### 5.1 推荐状态集合（可扩展）
- `Planned`：规划中（已申请预算/采购意向）
- `Ordered`：已下单（PO 已创建）
- `Received`：已到货（待验收/贴标）
- `InStock`：库存可用（可分配）
- `Reserved`：已预留（为某个申请锁定）
- `InUse`：使用中（已分配）
- `InTransit`：流转中（跨站点调拨）
- `UnderMaintenance`：维修中
- `Quarantined`：隔离（安全/合规原因，如涉密/疑似丢失）
- `Retired`：已退役（不再投入使用）
- `Disposed`：已处置（销毁/转卖/回收完成）
- `Lost`：丢失（可视为 Retired 的子类终态）

### 5.2 状态转换规则（关键）
- `InStock -> Reserved -> InUse`：分配路径（预留可选）
- `InUse -> UnderMaintenance -> InUse/InStock/Retired`：维修路径
- `InUse -> InStock`：归还验收通过
- `Any -> Retired -> Disposed`：退役与处置（必须有审批/记录）

### 5.3 企业级“强约束”示例（建议作为业务规则）
1. **维修中禁止分配**：`UnderMaintenance` 不能进入 `InUse`
2. **报废终态不可逆**：`Disposed` 不允许回到任何可用状态
3. **跨员工转移必须先归还**：`InUse(A员工)` 到 `InUse(B员工)` 必须经过归还/交接记录
4. **预留超时释放**：`Reserved` 超过 X 天未领取自动释放回 `InStock`
5. **隔离状态需安全审批**：`Quarantined` 只能由安全管理员解除

## 6. 关键业务流程（Enterprise Workflows）

### 6.1 采购与入库（Procure-to-Stock）
**目标**：保证从采购到入库全链路可追溯，可对账。

典型步骤：
1. 资产需求申请（Request）→ 预算/成本中心确认
2. 审批通过后生成采购订单（PO）
3. 到货登记（Received）：扫码录入 Serial / 贴资产标签
4. 验收清单：外观、配件、配置、质保信息
5. 入库完成：状态变为 `InStock`

企业级系统通常要能覆盖从采购到退役的端到端生命周期，融合财务、库存、合同、风险管理等要素。:contentReference[oaicite:5]{index=5}

---

### 6.2 员工入职配发（Onboarding Assignment）
**目标**：按岗位标准自动推荐设备组合，减少人工选择。

流程要点：
- 依据岗位类型（研发/销售/设计）匹配设备规格模板
- 自动检查库存可用性
- 生成交接单与领用确认（电子签名）
- 初始化安全策略（如 MDM、磁盘加密、资产水印）

---

### 6.3 临时借用（Borrow）与审批链（重点）
企业级“借用”通常与“长期领用”分离，核心差异在于：
- 有**预期归还日期**
- 有**超期提醒与升级通知**
- 更常出现**经理审批 + 资产管理员审批**的双层审批

**推荐审批链（可配置）：**
1. 申请人提交 BorrowRequest（理由、期限、资产类型）
2. 部门经理审批（确认业务必要性）
3. 资产管理员审批（确认库存与风险）
4. 发放资产（交接记录 + 状态变更 `InStock->InUse`）
5. 到期提醒（T-3/T-1）
6. 归还验收（清点配件、外观、数据擦除确认）
7. 归还成功 → 状态变更 `InUse->InStock`

**超期策略（常见企业规则）：**
- 超期 1 天：通知借用人
- 超期 3 天：抄送部门经理
- 超期 7 天：抄送 IT 主管/财务资产负责人

---

### 6.4 部门调拨与跨站点流转（Transfer）
企业级常见场景：A 城市库存富余，调拨到 B 城市。
- 必须引入 `InTransit` 状态
- 必须记录物流单号、交接人、签收人
- 到达后再入库 `InStock`

---

### 6.5 报修与维修（Repair / Maintenance）
维修流程的企业级要点：
- 维修工单（WorkOrder）必须关联资产
- 维修期间资产不可分配（状态 `UnderMaintenance`）
- 记录维修成本、备件更换、供应商
- 维修完成需质检与验收，再恢复到 `InUse` 或 `InStock`

EAM 强调维护管理与维修计划，通过维护策略延长资产寿命并降低成本。:contentReference[oaicite:6]{index=6}

---

### 6.6 员工离职清退（Offboarding Clearance）
离职清退是企业资产流失的最大风险点之一。

**强规则建议：**
- HR 发起离职流程时，自动生成“资产归还清单”
- 若仍有 `InUse` 状态资产未归还：离职流程不得关闭
- 支持例外流程：丢失/被盗（需报备与审批）

---

### 6.7 盘点与审计（Inventory Check & Audit）
企业级盘点能力：
- 支持周期盘点（季度/年度）
- 支持抽盘（高价值资产优先）
- 支持移动端扫码（二维码/RFID）
- 盘点结果：存在/缺失/位置变更/异常状态

盘点后的处理：
- 自动更新 `lastSeenAt`
- 缺失资产进入 `Quarantined` 或 `Lost` 流程
- 形成审计报告（盘盈盘亏）

---

### 6.8 报废与处置（Retire & Dispose）
报废不仅是状态变更，更是**合规与安全动作**：
- 数据安全擦除确认（硬盘销毁/安全擦除证明）
- 报废审批（管理员 + 财务/资产负责人）
- 处置方式：回收/转卖/销毁（保留凭证）
- 处置后资产状态进入终态 `Disposed`

## 7. 权限与组织模型（RBAC & Org）

企业级系统必须支持复杂组织结构：

### 7.1 组织层级
- 集团 / 子公司 / 部门 / 团队 / 成本中心 / 项目组
- 多站点：城市 / 园区 / 楼宇 / 楼层 / 仓库

### 7.2 常见角色（示例）
- `Employee`：普通员工（只能看自己的资产、提交借用/报修）
- `DeptManager`：部门经理（审批借用/领用、查看部门资产概览）
- `AssetAdmin`：资产管理员（分配/归还/盘点/报废）
- `Procurement`：采购专员（PO/合同）
- `Finance`：财务（折旧/台账/对账）
- `SecurityOfficer`：安全官（涉密资产、隔离/解隔离审批）
- `Auditor`：审计员（只读访问 + 报表）

### 7.3 职责分离（SoD）
企业级强要求：避免“一个人既申请又审批又报废”的风险
- 申请与审批不能同人
- 报废审批必须双人或财务复核

## 8. 企业级业务规则清单（可直接进 Spec）

### 8.1 强制唯一性
- `serialNumber` 全局唯一（硬件）
- `assetTag` 组织内唯一（或全局唯一）

### 8.2 并发控制
- 同一资产在同一时间只能有一个“当前持有关系”
- 分配/归还/状态变更必须事务化（避免重复分配）

### 8.3 状态前置校验（重要）
- 只有 `InStock` 才允许分配
- `UnderMaintenance/Quarantined/Disposed` 一律禁止分配

### 8.4 交接清单不可省略
- 分配时记录配件：电源、包、扩展坞、适配器
- 归还时必须核对配件与外观

### 8.5 审计日志不可删除
- 关键对象（Asset/Assignment/WorkOrder/Disposal）必须全量审计

## 9. 报表与指标（Dashboards & KPIs）

企业级系统通常要提供：
- 资产总量、按类型/部门/站点分布
- `InStock / InUse / UnderMaintenance / Retired` 状态占比
- 利用率：`InUse / (InUse + InStock)`
- 平均维修时长（MTTR）
- 故障率、维修成本、备件消耗
- 资产丢失率（年度）
- 软件许可合规率（超量/未授权安装）
- 折旧进度、净值、年度资产新增与处置

## 10. 集成与自动化（Integration & Automation）

### 10.1 常见集成点
- HR 系统：入职/离职/组织变更
- IAM/SSO：统一登录、权限同步
- 采购系统：PO、供应商、发票
- 工单系统：报修、ITSM 流程
- 资产发现（Discovery）：自动发现设备存在与配置（ITAM 常见能力）:contentReference[oaicite:7]{index=7}
- CMDB：资产与配置项关联（选做）

### 10.2 自动化规则（示例）
- 借用到期提醒与升级通知
- 维修超期提醒
- 库存不足预警（触发采购建议）
- 高价值资产盘点频率提升
- 安全策略自动下发（涉密资产必须加密）

## 11. 非功能要求（NFR）

### 11.1 性能与规模
建议以企业典型规模为目标：
- 资产 5 万~50 万条
- 并发在线用户 500~5000
- 常见查询 < 200ms（索引与缓存）

### 11.2 安全
- RBAC（最小权限）
- 审计日志完整性
- 导出权限控制（防止资产台账泄露）
- 数据擦除流程记录（报废合规）

### 11.3 可用性
- 关键流程（领用/归还/盘点）必须可用
- 重要操作需幂等（重试不重复扣库存/重复分配）

## 12. 典型边界场景（Edge Cases）

1. **资产丢失**：转为 `Lost`，生成责任追踪与审批记录  
2. **资产被盗**：安全流程与报案凭证上传  
3. **借用超期**：自动升级通知链路  
4. **跨部门争议**：同一资产归属不清（需归属组织字段与审批）  
5. **维修成本过高**：建议报废与财务复核  
6. **批量入库与批量分配**：事务一致性（全部成功或全部失败）  
7. **盘点发现位置变更**：位置更新必须审计记录
